# モーニングルーティン Web サイト 要件定義書

**版数:** 1.1
**作成日:** 2026-06-05
**ステータス:** 確定（実装着手）

---

## 1. 概要・目的

毎朝のルーティンを支援する個人専用の Web アプリケーションを構築する。
タスク管理・タイマー・習慣トラッカー・記録などを通じて、朝活を「続けられる」ことを最優先の目的とする。
スマートフォンのホーム画面に追加し、アプリのように日常使いすることを想定する。

### ゴール
- 毎朝の流れをワンタップで呼び出し、迷わず始められる
- 小さな達成を可視化し、継続のモチベーションを維持する
- スマホ単体・オフラインでも完結して動作する

---

## 2. 前提・制約

| 項目 | 内容 |
|------|------|
| 利用者 | 本人のみ（自分専用） |
| アカウント機能 | なし（ログイン不要） |
| サーバー | なし（バックエンドを持たない） |
| メイン端末 | スマートフォン（モバイルファースト設計） |
| データ保存先 | 利用端末のブラウザ内（localStorage / IndexedDB） |
| 配布形態 | 単一の Web サイトとして提供し、PWA としてホーム画面に追加 |

### 重要な制約事項（必読）
- **データはそのブラウザ内にのみ保存される。** ブラウザの「サイトデータ削除」や端末初期化を行うと記録は失われる。
  → 対策として **データのエクスポート（バックアップ）／インポート（復元）機能** を必須要件に含める。
- **予約通知は完全には保証できない。** 詳細は「7. 通知に関する仕様と制約」を参照。

---

## 3. 技術構成（方針）

- フロントエンドのみで完結する構成（HTML / CSS / JavaScript）
- **PWA 化**：マニフェスト（manifest）＋ Service Worker により、ホーム画面追加・オフライン起動に対応
- データ永続化：localStorage を基本とし、音声ファイル等の容量が大きいデータは IndexedDB を利用
- 外部サーバーへの通信は行わない（完全ローカル動作）

---

## 4. 画面構成（タブ切り替え UI）

下部タブバーによる画面切り替えを採用する（スマホで親指操作しやすい配置）。

**並び順（確定）：ホーム → 記事 → タスク → タイマー → 記録 → 設定**

| 順 | タブ | 役割 |
|----|------|------|
| 1 | ホーム | 今日の全体ダッシュボード（起床時刻・習慣・本日のルーティン） |
| 2 | 記事 | 朝活の恩恵・価値についての読み物（枠のみ） |
| 3 | タスク | タスクの管理 |
| 4 | タイマー | 各作業のタイマー実行 |
| 5 | 記録 | 履歴・統計・連続達成日数の可視化 |
| 6 | 設定 | BGM・通知・データ管理などの各種設定 |

---

## 5. 機能要件

### 5.1 ホーム（ダッシュボード）
- 今日の日付・時刻の表示
- **起床時刻の記録**：ワンタップで今朝の起床時刻を打刻、当日の記録として保存
- **習慣トラッカー**：チェック式の固定習慣を一覧表示し、チェックで完了。**初期は空の状態から開始**し、ユーザーが項目を追加（確定）
- **本日のルーティン**：呼び出し済みテンプレートの当日進行状況を表示
- 連続達成日数（ストリーク）のサマリー表示

### 5.2 ルーティンのテンプレート化
- 一連の朝の流れ（複数タスク／習慣のまとまり）をテンプレートとして保存
- **ワンタップ呼び出し**：テンプレートを選ぶと当日のタスク・習慣として一括展開
- テンプレートの作成・編集・削除

### 5.3 タスク管理
- タスクの**追加・削除**
- **進行度合い**：3 段階の状態（**未着手 / 進行中 / 完了**）で管理（確定）
- **完了マーク**の付与
- **日時更新**：作成日時・更新日時の記録と表示
- 当日分タスクと、テンプレート由来タスクの区別

### 5.4 タイマー
- **開始・停止（一時停止）・再開・リセット**
- **円形または横長の進捗ゲージ**による残量の可視化
- **タスクのドロップダウン選択**：実行中タスクを選び、計測対象として紐付け
- 時間設定（プリセット ＋ 任意指定）
- 終了時のアラート（音・バイブ・通知。制約は第7章参照）

### 5.5 記録
- タスク完了・習慣達成・起床時刻・タイマー実行の履歴を日次で保存
- カレンダー／一覧での振り返り
- **統計の可視化**：**連続達成日数（ストリーク）を主役**とし、補助指標として完了率・所要時間の推移をグラフ表示（確定）
- データの**エクスポート（バックアップ）／インポート（復元）**

### 5.6 記事（朝活の恩恵・価値）
- 記事を表示する**枠（レイアウト）のみ**を用意する
- 本文はユーザー自身が後から編集・差し込みできる構造とする
- 見出し・本文・区切りなど、読みやすい体裁を用意

### 5.7 設定
- **BGM・環境音のアップロード（mp3）**
  - 端末内の mp3 ファイルを取り込み、ライブラリとして保持（IndexedDB に保存）
  - 再生・停止、ループ、音量調整
  - タイマーやルーティン中に再生する音を自由に割り当て
- **通知設定**：通知の許可リクエスト、リマインダー時刻の登録・ON/OFF
- **データ管理**：エクスポート／インポート、全データの初期化
- テーマ・表示に関する設定（任意）

---

## 6. 非機能要件

### 6.1 デザイン
- **朝らしい柔らかさ（パステル系・ミント基調）** をテーマカラーとする（確定）
- モバイルファースト。片手・親指操作を前提としたタップ領域とレイアウト
- ライト基調を中心に、可読性とコントラストを確保

### 6.2 アニメーション
- **動的なアニメーションを UI に付与**する
  - タブ切り替え時のトランジション
  - タスク完了・チェック時のマイクロインタラクション
  - タイマーゲージの滑らかな動き
  - 達成時のフィードバック演出
- パフォーマンスと電池消費に配慮し、過剰にならない範囲で適用

### 6.3 PWA / オフライン
- manifest による「ホーム画面に追加」対応（アイコン・名称・起動時画面）
- Service Worker によるオフライン起動・アセットキャッシュ
- スタンドアロン表示（アドレスバーなしのアプリ風表示）

### 6.4 パフォーマンス
- 初回読み込み後はオフラインで即時起動
- 音声ファイルなど大容量データの保存・読み込みでも UI が固まらない設計

## 7. 通知に関する仕様と制約 ★重要

**方針：「できる範囲（ベストエフォート）」での予約通知を実装する。**

実装するもの：
- 通知許可のリクエスト（Notification API）
- リマインダー時刻の登録・管理
- アプリ起動中／バックグラウンド直後における時刻通知・タイマー終了通知
- Service Worker を介した通知表示

技術的な制約（合意事項）：
- スマホブラウザ、特に **iOS Safari は、アプリを長時間完全に閉じた状態での予約通知を確実には発火できない。**
- アプリを閉じている間にも確実に届く予約通知（プッシュ通知）には、通常 **サーバー側のプッシュ配信基盤が必要** となる。今回はサーバーを持たない方針のため対象外とする。
- 上記により、**「指定時刻に必ず鳴る」ことは保証できない**。アプリ起動中／直近の利用時を中心に動作する仕様とする。

> 将来的に確実な予約通知が必要になった場合は、プッシュ配信用の軽量バックエンド導入を別途検討する。

---

## 8. データ設計（概要）

ブラウザ内ストレージに以下を保持する（キー名は実装時に確定）。

- 設定情報（テーマ、通知設定、リマインダー）
- タスク（内容、状態、進捗、作成・更新日時）
- 習慣（固定習慣の定義と日次の達成状況）
- ルーティンテンプレート（タスク・習慣のまとまり）
- 起床時刻の記録（日次）
- タイマー実行ログ
- 記事本文（ユーザー編集分）
- 音声ファイル（mp3 バイナリは IndexedDB に保存）
- 統計算出用の履歴データ

エクスポート時は、音声以外のデータを JSON 形式でまとめて書き出す。

---

## 9. スコープ外（今回は実装しない）

- 複数ユーザー・アカウント管理・ログイン
- 端末間のデータ同期
- サーバー連携・クラウドバックアップ
- 朝活記事の本文執筆（枠のみ提供）
- 外部サービス連携（カレンダー・天気 API など）

---

## 10. 残課題・確認事項（すべて確定済み）

| 項目 | 決定 |
|------|------|
| タブの並び順 | ホーム → 記事 → タスク → タイマー → 記録 → 設定 |
| タスクの進行度 | 3 段階（未着手 / 進行中 / 完了） |
| 習慣トラッカーの初期項目 | 空から開始（ユーザーが追加） |
| 統計の主役指標 | ストリークを主役、完了率・所要時間を補助 |
| パステル基調色 | ミント系 |
